# i18n Rules

Glyph ships five locales: `en`, `de`, `es`, `fa`, `zh`, under `src/locales/<locale>/<namespace>.json`. English is the source of truth; the others must mirror it exactly.

- **No hardcoded user-visible strings.** Anything a user reads (labels, buttons, toasts, tooltips, aria-labels, error messages, empty states) goes through `useTranslation(namespace)` / `t()`, never as a literal in JSX or Rust.
- **New keys land in all five locales in the same commit.** Adding a key to an English namespace means adding the translated value to `de`, `es`, `fa`, and `zh` at the same time, not as a follow-up. `src/locales/localeParity.test.ts` fails the suite when any locale is missing (or has extra) keys relative to English, so drift cannot merge.
- **Native menu items are translated through the `menu` namespace.** The Rust menu is built with English defaults and re-labeled at runtime: `useNativeMenuLabels` pushes the flat `menu` bundle to the `set_menu_labels` command, which writes each field of the `MenuLabels` struct onto its `MenuItemRefs` entry. Adding a menu item therefore touches, together: the item in `menu_runtime.rs`, a field in `MenuLabels` **and** `MenuItemRefs` **and** `apply_menu_labels`, and a `menu.json` key (camelCase of the Rust field) in all five locales. Missing any of these leaves the item permanently English.
- **Palette entries live in the `commands` namespace**; keep the palette title and the menu label consistent in meaning (they may differ in wording, e.g. "Plugins…" in the menu vs "Manage Plugins…" in the palette).
- Interpolations use i18next syntax (`{{name}}`); keep placeholder names identical across locales.
- Plugins bring their own strings via `ctx.registerTranslations`; those are exempt from the five-locale rule (plugin authors choose their locales).
